# B 站视频文字稿：我为什么不用 AI 写作网站，而是做了一个 Claude Code 小说 Skill

大家好，今天想聊一个我最近自己写的项目。

它是一个 AI 长篇小说创作引擎，名字暂时叫 `novel-engine`。

但我不太想把它叫成“AI 小说生成器”。

因为现在网上已经有很多 AI 写小说的网站了，点进去填几个角色、填几个设定、选一下风格，然后点生成。

我自己之前也用过不少类似产品，国内国外都有。用下来最大的感觉是：它们不是完全不能写，而是很难真正按我的工作流写。

很多产品的问题是这样的：

第一，模板很固定。

它会让你填世界观、角色、简介、大纲，但字段都是它规定好的。你想多加一套关系网，或者想把伏笔、时间线、角色状态单独管理，就很别扭。

第二，导入很麻烦。

如果已经有一本旧书、有几十章正文、有一个参考作品，想让它拆书、总结、提炼节奏，再基于这个写同人、番外、续写，很多网页工具都不太顺。

第三，工作流太封闭。

它给你按钮，你只能按它预设的流程走。但长篇创作不是一个固定表单。今天可能要改角色设定，明天要补一章番外，后天要把第 80 章之后的大纲整体重排，再过几天又想导入一本文做参考蓝图。

这类事情如果放在一个固定网页里，反而会越来越难用。

所以后来我想，为什么要重新做一个 AI 写作前端？

Claude Code、Codex 这类 agent，本身已经会读文件、改文件、跑命令、分析长文本、维护项目结构、生成大纲、修 bug。

那是不是可以不造一个新的封闭写作 App，而是把小说创作变成一个 agent 能直接操作的工程项目？

这就是 `novel-engine` 的核心思路。

它不是一个网页写作器，而是一个 agent-native 的长篇写作 harness。

中文说，就是一个“面向 Agent 的长篇创作 Skill”。

## 为什么 256K 上下文还不够

很多人现在会觉得，大模型上下文都 128K、256K 了，那写长篇小说是不是直接把全文塞进去就行？

一开始我也这么想过。

但真写起来会发现，不是这么回事。

长篇小说不是一大坨文本。它是一个持续变化的结构系统。

比如一个角色，前 20 章是什么状态，第 50 章关系变了，第 100 章受过伤，第 120 章知道了某个秘密。这些东西不是简单“出现过”就行，而是要知道在当前章节，它到底处于什么状态。

再比如伏笔。

第 7 章埋了一个照片，第 40 章提了一次，第 100 章才开始回收。模型如果只是读一堆全文，它未必知道这个东西在当前章应该怎么用。

还有关系。

两个人是刚认识、暧昧、同盟、恋人，还是互相试探？这些关系会变，而且变化有时间点。

所以问题不只是上下文长度，而是上下文没有结构。

把几十万字全塞进去，成本很高，重点也散。模型可能看过了，但它不知道本章最应该遵守哪几条。

所以我做的不是“更长 prompt”，而是“上下文编译”。

每一章生成之前，系统会把这一章需要的信息整理出来。

比如：

- 本章大纲
- 前情摘要
- 邻近章节实况
- 交叉引用章节
- 角色卡
- 当前角色状态
- 动态关系网
- 未解伏笔
- 世界书设定
- 拆书提炼出来的蓝图
- 可选的向量索引和结构化事实

然后把这些编译成本章 prompt。

每一章就像一个小型 subagent。它不是直接吃整本书，而是拿到自己这章真正需要的上下文。

## 项目结构

一本小说就是一个文件夹：

```text
projects/某本小说/
├── outline.json
├── lorebook.json
├── chapters/
├── state/
│   ├── state.json
│   └── profiles.json
├── book.md
└── book.txt
```

`outline.json` 负责世界观、人物、章节大纲。

`state.json` 负责记录已经写完哪些章节，每章摘要是什么。

`profiles.json` 负责角色档案、状态快照、伏笔、关系网。

`lorebook.json` 是世界书，类似关键词触发的设定库。

`chapters/` 下面就是每一章正文。

然后还有一个 `SKILL.md`。

这个文件是给 Claude Code 或 Codex 看的。它告诉 agent：如果用户要生成章节，应该先检查什么，运行什么命令，生成后怎么检查，什么时候合并，什么时候更新档案。

所以它有两层：

第一层是 engine，也就是 Python 后端和 CLI。

第二层是 skill，也就是 agent 操作说明。

我自己真正喜欢的是第二层。

因为这样之后，我不是在操作一个死板的网页，而是在和一个成熟 agent 一起维护一整本小说工程。

可以直接说：

“帮我生成第 12 章。”

“帮我预览第 50 章会注入哪些上下文。”

“帮我把第 3.5 章作为番外插进去。”

“帮我检查最近 10 章有没有角色状态倒退。”

“帮我把这本文导入，提炼成参考蓝图。”

这些事情放在固定网页里会很麻烦，但放在文件系统 + CLI + Skill 里，就很自然。

## 和传统 AI 写作产品的区别

这里有个很重要的区别。

传统 AI 写作产品，更像一个封闭工作台。

你在它给你的页面里填设定，点按钮，生成章节。

而 `novel-engine` 更像一个开放的小说工程目录。

所有东西都是文件：

大纲是文件。

角色卡是文件。

伏笔是文件。

章节是文件。

摘要是文件。

Skill 也是文件。

Claude Code 可以直接读，直接改，直接运行命令。

如果想改一个角色设定，不需要等产品支持某个按钮。直接改 `profiles.json`。

如果要插番外，章节 ID 可以写成 `3.5`。

如果要续写旧书，就把 txt 导入，让系统切章，生成摘要。

如果要参考某本书的节奏，就让 extractor 拆书，提炼蓝图。

如果要清理 AI 输出里的字数统计、元注释，就跑清洗脚本。

它不是一个“漂亮但固定”的写作界面，而是一套可以被 agent 操作的长篇写作基础设施。

## 真实验证

目前我用它做了一个真实长篇项目。

规划是 380 章，现在已经生成到 152 章。

单章大概 3000 到 8000 字。

当然，它还不是完美的。AI 还是会犯错，有时候会出现重复解释，有时候会有 prompt 残留，有时候要人工清洗。

但它比较有意思的是，长篇一致性比我之前直接 prompt 写要好很多。

因为每一章都能拿到当前角色状态、关系网、伏笔和邻近章节摘要。

举个例子。

写第 151 章时，系统不是简单让模型“继续写”。

它会告诉模型：

前面发生过什么。

哪些角色现在是什么状态。

哪些关系已经成立。

哪些伏笔还没回收。

本章需要呼应哪些旧章节。

附近几章实际写成了什么样。

本章不能提前剧透后面哪些东西。

这就是我说的上下文纪律。

长篇创作最怕的不是模型文笔差，而是它没有纪律。

今天忘了昨天谁受伤，明天忘了谁已经告白，后天把一个已经解决的伏笔又当新线索写。

这个项目就是试图给模型加一层工程化纪律。

## 不只是原创小说

它能做的也不只是原创小说。

我觉得它可以用于很多长文本创作任务。

比如同人创作。

导入原作资料，继承人物和世界观，再写新剧情。

比如初稿扩写。

只有世界观和大纲，也可以先批量生成一个可修改初稿。

比如想法成书。

脑子里有设定、有角色、有几条线，但没时间一章章敲，可以先让系统生成结构化初稿。

比如中外小说互译。

逐章导入，保持人名、术语、关系和伏笔一致。

比如角色设定改写。

把某个角色关系改掉，然后重写相关章节。

比如番外补章。

在已有正文中间插一个自己想看的番外，还要接上前后章。

甚至可以做长篇 IP bible。

维护角色、地点、组织、道具、时间线、关系和冲突记录。

所以它本质上不是“写小说”这么窄。它是一个长文本上下文管理工具。

## 为什么不先做前端

为什么不先做前端？

因为之前真的做过。

做完发现，前端太容易消耗注意力。

结构一变，接口要改，状态要改，组件要改，页面要改。

vibe coding 的 token 全花在 CSS、水合错误、前端状态管理上。

结果核心问题，也就是“怎么让 AI 写长篇不失忆”，反而没时间打磨。

所以这次反过来。

先不做前端。

先把核心能力做成 CLI。

再把 CLI 组织成 Skill。

让 Claude Code 或 Codex 这种成熟 agent 来当操作界面。

等工作流稳定了，再考虑前端。

现在越来越觉得，对这种 AI 工具来说，最好的 MVP 不一定是一个最小前后端产品，而是一个能被 agent 稳定调用的后端技能。

因为 agent 本身已经是界面了。

## 总结

如果要一句话总结这个项目，我会这么说：

`novel-engine` 是一个 agent-native 的长篇写作 harness。

它不重新造一个固定模板的写作网站，而是把小说项目暴露成文件、命令和 Skill，让 Claude Code / Codex 直接接管创作流程。

它解决的不是“怎么让 AI 写一段文字”，而是：

怎么让 AI 在一部长篇里持续记住世界观、角色状态、关系变化和伏笔。

目前代码已经整理成开源仓库。

README 里放了第一章样章，也放了脱敏后的 prompt 结构示例。

完整正文不会放仓库里，因为后面还要走小说平台。

如果也遇到过 AI 写长篇越写越乱的问题，可以看看这个思路。

不是换一个更大的上下文窗口，而是给模型一套更好的上下文工程。
