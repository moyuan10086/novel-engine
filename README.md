# novel-engine

面向中文长篇小说和长文本创作的 LLM 写作引擎，也可以作为 Claude Code / Codex / 其他 coding agent 的写作 Skill 使用。它不是单次 prompt 脚本，而是一套围绕“长篇连载不失忆”设计的本地 CLI + Skill 工具：用大纲、章节摘要、角色档案、伏笔、关系网、世界书和邻近章节实况，为每一章组装更合适的上下文。

它不只适合原创小说，也适合所有需要“长程一致性 + 结构化上下文”的创作任务：同人创作、初稿扩写、想法成书、中外小说互译、角色设定改写、番外补章、旧作续写、名著续写、世界观资料整理和长篇 IP 文档维护。

> 当前定位：开发者与作者的本地创作工具。既可以直接用 CLI 运行，也可以把 `SKILL.md` 交给 agent，让 agent 按固定工作流接管长篇创作。

## 为什么做这个

大模型上下文窗口越来越长，但把几十万字全部塞进去并不等于能写好长篇。长篇小说真正容易坏掉的是：

- 角色状态倒退：境界、关系、伤势、身份前后不一致
- 伏笔遗忘：前面埋过的线后面完全消失
- 设定漂移：世界规则越写越松
- 番外难插：插入中间章节时接不上前后文
- 长篇节奏失控：每章都像独立短篇，没有卷纲推进

novel-engine 的思路是把小说项目工程化：每章生成前先编译上下文，每章生成后再记录摘要和状态，让模型写作时拿到“本章真正需要的信息”。

### 产品哲学：不是封闭写作台，而是 agent-native harness

传统 AI 写作产品更像一个封闭工作台：在产品给定的界面里填写设定、选择模板、点击按钮、生成章节。这个范式适合快速上手，但当创作流程变复杂时，固定字段、固定模板、固定按钮和困难的导入流程会变成限制。

novel-engine 更像一个 **agent-native writing harness**：它把小说变成一个文件系统项目，然后让 Claude Code / Codex 这类成熟 agent 去操作它。

在这个范式里：

- 大纲是文件
- 角色卡是文件
- 伏笔和关系网是文件
- 章节正文是文件
- 世界书是文件
- Skill 也是文件

agent 可以直接读取、修改、运行命令、检查状态、重排大纲、维护角色档案、生成章节和清洗读者稿。项目不试图重新造一个弱版写作 agent，而是复用已有 coding agent 的文件操作、规划、执行和迭代能力。

### 256K 上下文仍然不够

现在的大模型动辄 128K、256K 甚至更长上下文，但长上下文不是长篇小说的完整解法。小说不是一段平铺文本，而是一个持续变化的结构系统：人物状态会变，关系会变，伏笔有埋设和回收，世界观有规则边界，读者还会记得几十章前的一句对白。

直接把大量正文塞进上下文，会遇到几个问题：

- 成本高：很多 token 花在本章并不需要的信息上
- 重点散：模型在海量文本里抓不到本章真正关键的状态
- 不可审计：出了错很难判断是漏了哪条设定
- 难维护：角色状态、关系、伏笔都混在自然语言里

所以百万字级别的小说生成，更需要一套系统化的 skills / harness，而不是只依赖一个超长 prompt。

### Harness：把长文本创作变成可管理的上下文系统

本项目把每一章、每一节或每一次改写任务都看成一个“小型 subagent 任务”：它不直接吃整本书，而是拿到一组经过整理的上下文块。

这些上下文块包括：

- 世界观与规则
- 本章大纲和关键节拍
- 前情摘要与邻域实况
- 交叉引用章节摘要
- 角色卡和动态状态快照
- 当前关系网
- 未解伏笔
- 世界书条目
- 可选语义索引与结构化事实
- 拆书/蓝图提炼出的节奏参考

换句话说，novel-engine 做的是一种“上下文编译”：把原始创作工程里的世界观、设定、索引、角色卡、动态关系、蓝图、拆书结果和结构化知识，编译成本次写作最需要的 prompt。这个过程不能保证模型永远不犯错，但实践中对 100+ 章连续生成、长篇翻译、续写和番外补章的稳定性帮助很明显。

### 适用场景

novel-engine 更适合这些需要长期记忆和结构控制的创作工作流：

- **同人创作**：导入原作资料，继承世界观和人物关系，写全新剧情
- **初稿书写**：把已有大纲快速扩展成可修改的章节初稿
- **想法成书**：把脑内设定、角色和情节线变成完整项目
- **中外小说互译**：导入原文切章，逐章翻译并保持术语、人名和设定一致
- **角色设定改写**：修改角色档案后重写相关章节
- **番外补章**：在已有正文中插入自己想看的番外，并衔接前后章
- **名著/旧作续写**：从旧文本中提炼蓝图，再进行续写或再创作
- **长篇 IP 资料管理**：维护世界观、人物关系、伏笔、时间线和设定索引
- **系列文档写作**：把一组共享设定的长文档拆成可追踪、可续写、可检查的项目

### 它也可以是一个 Skill

novel-engine 不只是命令行工具，也可以被包装成一个 agent skill。

在这种用法里，使用者不需要记住所有命令，而是让 Claude Code / Codex 读取 [SKILL.md](SKILL.md)，再用自然语言下达任务：

- “给这个项目生成第 12 章”
- “检查当前进度”
- “预览第 50 章会注入哪些角色档案”
- “把第 3.5 章作为番外插入”
- “合并出读者版”
- “检查最近 10 章有没有状态倒退”

Skill 的价值在于把一套复杂创作流程固定下来：什么时候看 `outline.json`，什么时候读 `state.json`，什么时候更新 `profiles.json`，什么时候合并，什么时候清洗读者稿。这样 agent 操作的不是一堆零散脚本，而是一套稳定的长篇小说生产规范。

换句话说，本项目有两层形态：

- **Engine**：`src/` 里的 Python 后端和 CLI
- **Skill**：`SKILL.md` 定义的 agent 操作流程

这也是为什么项目先做 CLI：CLI 是最容易被 agent 稳定调用的接口，而 Skill 是把这些接口组织成工作流的说明书。

### 为什么先做 CLI + Skill，而不是前端

早期原型也尝试过带前端的小说创作器，但这个阶段最消耗精力的并不是 UI，而是核心生成策略。

前端 MVP 在这类项目里会带来几个问题：

- 结构一变，前端状态、路由、组件和接口都要跟着变
- vibe coding 的 token 很容易花在 CSS、布局、水合错误、组件状态上
- prompt 策略和数据结构还在快速变化时，前后端一起改会非常脆
- 为了追求“最小可用前后端”，反而拖慢最核心的上下文工程迭代

所以 novel-engine 先选择纯后端 + CLI + Skill 的形态。CLI 更适合快速验证：

- 改 prompt 只需要改模板或 Python 模块
- Claude Code / Codex 可以直接充当操作界面
- workflow 可以沉淀成 Skill，而不是过早固定成 UI
- 等核心能力稳定后，再包前端也不迟

对 AI 写作工具来说，最好的 MVP 不一定是“最小前后端”，而可能是“可被 agent 稳定调用的后端技能”。

### 不只是写小说

虽然项目从中文长篇小说场景里长出来，但核心能力其实是“长程创作上下文管理”。只要任务需要在很多章节、很多人物、很多设定之间保持一致，就可以复用同一套方法。

它可以被理解成一个面向长文本的 skills harness：

- 写原创长篇：按卷纲、章纲和角色档案持续生成
- 写同人/番外：继承已有设定，插入新剧情
- 做初稿扩写：把设定卡和大纲快速扩成可编辑正文
- 做跨语言翻译：保持人名、术语、口吻和伏笔一致
- 做设定改写：替换人物关系或世界规则后重写相关章节
- 做 IP bible：维护角色、地点、组织、道具、时间线和冲突记录

本质上，它解决的不是“调用哪个模型写一段话”，而是“如何让模型在一个持续变化的创作工程里保持上下文纪律”。

## 核心能力

- **多项目管理**：每本小说独立放在 `projects/<name>/`
- **章节调度**：支持 `strict` / `window` / `parallel` 三种生成模式
- **断点续传**：已完成章节记录在 `state/state.json`，重复运行会跳过
- **角色档案**：静态设定 + 稀疏快照，按章节生成有效状态卡
- **伏笔与关系网**：写作时自动注入当前未解伏笔和已建立关系
- **邻域实况**：注入当前章前后若干章摘要，减少接续断裂
- **交叉引用**：大纲中 `chXX` / `第XX章` 可触发对应章节摘要注入
- **番外补章**：支持 `3.5` 这类浮点章节 ID，自动取前后章节上下文
- **世界书**：关键词激活设定条目，只注入相关 lore
- **导入续写**：把已有 `.txt` 切章入库，再续写或插入番外
- **参考蓝图**：从参考文本提炼节奏、桥段、结构和写法，不复制原文
- **多 Provider**：OpenAI 兼容接口 / Anthropic 原生接口
- **读者稿清洗**：清理字数统计、扩写说明、缺失章节占位等工程痕迹

可选增强：

- ChromaDB 向量检索
- 章节事实提取
- 冲突检测
- YAML 题材/风格模板

第1章 配角阅读器开机

林夜把身份证按在便利店的扫描器上时，凌晨两点十七分，屏幕跳出“欢迎光临”的绿字。他十七岁了，按理说该买点有仪式感的东西，比如蛋糕或者限量汽水，可他只想买啤酒。不是为了喝醉，只是想在生日这一天做点剧本里绝对不会安排的事——一个二手钢琴店老板的儿子，半夜买啤酒，够俗气，够不体面。

收银台后面没人。

货架灯亮着，冷柜嗡嗡作响，空调风吹得塑料袋哗啦啦响，可就是没人。林夜等了十秒，又等了二十秒，最后干脆自己绕到柜台后面拿了罐冰啤酒。扫描器自动跳出金额，他把钱塞进零钱盒，动作像做贼，却又莫名理直气壮。

「这算不算偷啊？」他心里嘀咕，「还是说便利店今晚集体罢工？」

正要走，身后传来一个声音，不疾不徐，像在念一篇写得太长的散文。

「啤酒配生日，口感会涩。试试这个。」

林夜猛地回头。一个戴着旧鸭舌帽的中年男人不知什么时候站在了饮料区尽头，手里举着一罐明显过期了的橙味可乐，罐身凹陷，标签都快掉下来了。他笑的时候眼角皱纹很深，像常年熬夜看书留下的痕迹。

林夜下意识后退半步。「店员呢？」

「今晚空班。」男人把可乐递过来，「我替他们请你喝。生日快乐，林夜。」

他叫出了名字。

林夜接过可乐时指尖冰凉。他低头看罐底的生产日期，果然是半年前的玩意儿。可奇怪的是，他没觉得害怕，反而有种荒诞的熟悉感，像小时候读过的某本冷门推理小说，凶手总在最不该出现的地方递给主角一杯毒酒。

「你认识我？」

「比你想象中久。」男人靠在货架上，帽檐压得很低，只露出半张脸，「我问你一个问题。如果你发现自己一直被一本书写着，会怎么办？」

林夜差点被可乐呛到。他擦了擦嘴，忍不住笑出声，笑声在空荡荡的店里显得格外突兀。

「叔叔，你是来拉我入教的吗？还是说这是什么整蛊节目？」

男人没笑，只是静静看着他。那眼神像在看一本翻了太多遍的书，页边都毛了。

「书里的角色，最可怜的不是知道自己是角色，而是以为自己能改结局。」他顿了顿，又补了一句，「可乐过期了，喝的时候记得别太快。」

说完，他转身往店后面走。林夜追了两步，却发现那条窄窄的过道尽头只有一扇标着“员工休息室”的门。门没关严，里面黑乎乎的，什么都没有。他推开门，灯自动亮了，空无一人。

啤酒还握在手里，可乐罐却忽然变得沉甸甸的。

林夜站在原地发了会儿呆，最后把两罐饮料一起塞进塑料袋，推开玻璃门走了出去。东海市的夜风带着九月的潮气，路灯把他的影子拉得很长。他走着走着，忽然觉得身后好像有人在注视，可每次回头都只看到空荡荡的街道和闪烁的红绿灯。

回到家已经快三点了。父母都睡了，客厅茶几上放着一个没拆封的蛋糕，旁边是张卡片，妈妈娟秀的字写着“十七岁快乐”。林夜盯着那行字看了半天，忽然觉得哪里不对劲，却又说不上来。

他洗了把脸，躺在床上刷手机。指尖无意识地点开应用商店，想看看有没有新推送，却发现桌面多了一个没见过的图标——灰白底色，没有名字，只有一个简陋的书本图案，像临时用画图工具画的。

林夜点进去。

屏幕闪了一下，弹出提示框：【配角阅读器已激活。冷却时间计算中……】

「什么鬼？」他低声骂了一句，以为是哪个病毒软件。可下一秒，界面自动跳转，跳出一个搜索栏，下面写着一行小字：【输入目标姓名或指向拍摄，即可读取人物卡。注意：频繁使用将触发剧本系统反制措施。】

林夜盯着那行字看了半天，忽然觉得喉咙发干。他试探性地在搜索栏里输入了自己的名字。

屏幕卡顿了两秒，然后弹出一张卡片，像极了那些廉价手游的角色面板，只是文字冷冰冰的，没有任何花哨特效。

【配角A·林夜】
【年龄：17】
【身份：东海三高高二学生，二手钢琴店老板之子】
【剧本作用：2031年9月10日下午班级冲突中被周明朗当众羞辱，引出气运转折】
【剧本场次编号：147】
【当前状态：未破，温吞内向外壳下藏着不合时宜的精明】
【备注：适合作为早期垫脚石，建议保持低调直至被打脸】

林夜盯着最后那行“适合作为早期垫脚石”看了足足十秒，然后“呵”地笑了一声，笑完之后发现自己手指在抖。

场次编号147。这什么玩意儿？把人生切成镜头编号？这是哪门子元小说烂梗？他忽然想起小时候看的那些冷门推理小说，里面总有侦探在最后一页发现自己其实是书里的角色，然后发疯。可现实比小说更恶心，它连发疯的机会都不给，直接给你发了个APP。

他把手机扔到一边，盯着天花板。生日凌晨，空无一人的便利店，过期可乐，还有那个戴鸭舌帽的男人问的问题——如果你发现自己一直被一本书写着，会怎么办？

「我会先把作者揍一顿。」林夜对着空气说，声音低得只有自己听得见，「然后把这破剧本烧了。」

可笑的是，他说完这句话后，手机忽然震动了一下。屏幕亮起，那灰白图标又跳了出来，自动弹出新提示：

【检测到剧本偏离倾向。建议保持原有人设。首次读取冷却时间：30分钟。期间感知将出现钝化，请谨慎使用。】

林夜盯着那行字，忽然觉得太阳穴突突直跳。他想起刚才在便利店，那个男人递可乐时的表情——像在看一个注定要摔得很惨的角色，却又带着一点点……怜悯？

他坐起身，把手机反扣在桌上。窗外天色已经开始发白，远处有早班地铁驶过的声音。十七岁生日，就这么过去了。他原本以为会是个普通的生日，顶多被同桌小胖调侃两句“又老了一岁”，然后继续过那种温吞的校园生活。可现在，他忽然意识到，自己的每一次心跳、每一次呼吸，可能都被写成了带编号的场次。

147。

这个数字像根刺，扎在脑子里怎么都拔不出来。他闭上眼，又睁开，脑海里反复回放男人最后说的那句话。

“书里的角色，最可怜的不是知道自己是角色，而是以为自己能改结局。”

「那我就偏要试试。」林夜对着天花板轻声说，声音里带着点赌气的狠劲儿，「看看这破系统能把我怎么样。」

他重新拿起手机，把那罐过期可乐从塑料袋里翻出来，打开喝了一口。味道果然又甜又涩，像命运开的一个烂玩笑。可他还是喝完了，然后把空罐子捏扁，扔进垃圾桶。

手机屏幕还亮着，配角阅读器安静地躺在桌面，像个刚被拆封的潘多拉盒子。林夜盯着它看了很久，最后伸出手指，在搜索栏里又输入了自己的名字。

这次没有立刻弹出卡片，而是跳出一个红色警告：

【冷却中。感知钝化已启动。建议休息。】

林夜盯着那行字，忽然笑了起来。笑声很轻，却带着点破罐子破摔的痛快。

林夜在心里想。

## 安装

需要 Python 3.11+。

```bash
git clone https://github.com/<your-name>/novel-engine.git
cd novel-engine

uv sync
cp .env.example .env
```

完整安装：

```bash
uv sync --all-extras
```

按需安装：

```bash
uv sync --extra dev        # pytest
uv sync --extra tokens     # tiktoken
uv sync --extra vector     # chromadb
uv sync --extra anthropic  # Anthropic provider
uv sync --extra templates  # YAML templates
```

`.env` 示例：

```env
OPENAI_API_KEY=sk-xxx
OPENAI_API_BASE=https://api.openai.com/v1
MODEL=gpt-4o
USE_STREAM=1
CONCURRENCY=3

# optional
PROVIDER=openai
ANTHROPIC_API_KEY=
EXTRACT_FACTS=0
MAX_TOKENS=8000
TEMPERATURE=0.85
```

## 快速开始

创建项目：

```bash
uv run python -m src.cli create "我的小说" --genre "都市悬疑"
uv run python -m src.cli default "我的小说"
```

编辑：

```text
projects/我的小说/outline.json
```

生成章节：

```bash
uv run python -m src.dispatcher --project "我的小说" --only 1 --force
```

批量生成：

```bash
uv run python -m src.dispatcher --project "我的小说" --mode window --window 1 --concurrency 3
```

合并读者稿：

```bash
uv run python -m src.merger --project "我的小说"
```

输出：

```text
projects/我的小说/book.md
projects/我的小说/book.txt
```

清洗已导出的文本：

```bash
uv run python scripts/clean_reader_text.py "projects/我的小说/book.txt" --max-chapter 100
```

## 常用命令

项目管理：

```bash
uv run python -m src.cli list
uv run python -m src.cli info "我的小说"
uv run python -m src.cli context "我的小说" --chapter 12
```

角色档案：

```bash
uv run python -m src.profiles_cli char add "我的小说" "主角" --json '{"identity":"高中生"}'
uv run python -m src.profiles_cli snap add "我的小说" "主角" --at 10 --json '{"状态":"受伤"}'
uv run python -m src.profiles_cli fore add "我的小说" "F001" --at 3 --what "旧照片里的人影" --imp high
uv run python -m src.profiles_cli context "我的小说" --at 12
```

世界书：

```bash
uv run python -m src.cli lorebook add --project "我的小说" \
  --id "old_library" \
  --title "废弃图书馆" \
  --keys "图书馆,旧书,借阅卡" \
  --content "废弃图书馆是剧本系统的异常节点。"

uv run python -m src.cli lorebook test --project "我的小说" --text "主角进入废弃图书馆" --chapter 50
```

导入旧文：

```bash
uv run python -m src.importer --project "续写本" --src "source.txt" --skip-summary
```

提炼参考蓝图：

```bash
uv run python -m src.extractor --src "reference.txt" --output "blueprint.md" --sample 30
```

## 上下文机制

普通章节通常包含：

- 世界观和本章大纲
- 已完成章节摘要
- 本章前后邻域摘要
- 大纲交叉引用章节摘要
- 当前角色状态卡
- 当前关系网
- 当前未解伏笔
- 命中的世界书条目

插入章/番外额外包含：

- 前章摘要
- 后章摘要
- 紧邻前章原文末尾
- 紧邻后章原文开头

这套机制不能保证模型永不犯错，但能显著减少“长篇后期忘设定”的概率，也让问题更容易被定位和修正。

## 公开仓库内容边界

为了避免泄露私有创作项目、影响平台首发/签约，公开仓库建议只包含引擎代码、脱敏示例和少量样章。

- 不提交完整正文到 GitHub
- 不公开 `projects/<作品名>/chapters/`
- 不公开完整 `book.txt` / `book.md`
- 不提交 `.env`
- README 可放少量样章或上下文报告，但不要放完整书稿、后续章节剧透或完整真实 prompt
- 是否影响平台首发/签约，以具体平台后台规则和合同为准

本仓库的 `.gitignore` 默认会忽略本地小说项目中的正文、状态、合稿和向量库。
